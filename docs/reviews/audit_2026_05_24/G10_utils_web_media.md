## Cross-cutting notes

### SSRF posture (current)
The primary `url_fetcher.py` is now robust: explicit `_BLOCKED_NETWORKS` (incl. `::ffff:0:0/96`, CGN `100.64/10`, Alibaba `100.100.100.200/32`, AWS/Azure/GCP `169.254/16`), `ipv4_mapped` unwrap, scheme allowlist enforced on the initial URL **and every redirect target**, `allow_redirects=False` with a manual ≤5-hop loop, circular-redirect detection, a connect-time `_SSRFSafeResolver` (DNS-rebinding TOCTOU guard) receiving AF_UNSPEC (both A+AAAA checked — verified), `trust_env=False` (ignores `HTTP_PROXY` so the proxy IP can't bypass the resolver), magic-byte binary rejection, and exact-MIME matching. This is a strong, layered posture.

Residual SSRF themes (all MEDIUM/LOW): (1) the client fallback's inline backstop blocklist is weaker than canonical and only single-source-of-truth would fix it (M1); (2) decimal/octal/hex IP literals depend on libc getaddrinfo behavior — safe on this Windows host, needs prod-OS verification (M3); (3) GitHub-branch JSON/README reads bypass the byte cap (M2).

The Go service (`main.go`) independently enforces SSRF: scheme check, 8 KB URL cap, 20-URL batch cap, metadata-host blocklist, **and** a connect-time `ssrfSafeDialContext` that re-resolves and dials validated IP literals (defeats rebinding), binding to `127.0.0.1` only. So even the `_fetch_via_service` path is double-guarded (Python pre-check + Go dial-time check). H27's hard-fail-on-import-error is the correct posture and is implemented.

### yt-dlp / media
`from_url` enforces http(s) on input, delegates SSRF to the shared `_is_private_url` (with a fail-closed minimal fallback), and re-validates the post-extract filename: rejects empty, `-`-prefixed (ffmpeg flag injection), non-http(s) stream URLs, and confirms download paths stay inside `temp/` via `resolve()`+`relative_to`. `max_filesize=300 MiB` caps disk-fill DoS. The remaining items are the relative-`temp/` anchor (L: CWD dependence) and `search_source` not re-validating post-extract (L: depends on callers routing through `from_url`).

### fast_json
H29 fully fixed — orjson branch now forwards `default=`, maps `sort_keys`/`indent` to OPT flags, and raises `TypeError`/`NotImplementedError` on unsupported kwargs; stdlib branch mirrors it and forces `allow_nan=False` for cross-backend file compatibility. Residual: `json_dumps_bytes` lacks `OPT_NON_STR_KEYS` so it diverges from `json_dumps` on int-keyed dicts (L). Note: `cogs/ai_core/storage.py` defines its OWN local `json_dumps` (storage.py:24) that is unrelated to this module and routes `ensure_ascii=False/indent` to stdlib — out of this scope but confirms callers of the shared helper are unaffected.

### Dependency currency (verified at runtime; cutoff Jan 2026)
- **aiohttp 3.13.5** — current 3.x; resolver/connector signatures match the code's assumptions (verified `AbstractResolver.resolve(self, host, port=0, family=AF_INET)` and `TCPConnector(family=0)`). No known unpatched CVE applies to this usage at the audit date; `trust_env=False` mitigates proxy-env classes. Recommend keeping ≥3.13.
- **yt-dlp 2026.03.17** — recent rolling release; yt-dlp ships frequent extractor/security fixes, keep auto-updated. No code-level issue.
- **orjson 3.11.8** — current; behavior (UTF-8-only, NaN rejection, OPT flags) matches the wrapper. Verified `OPT_NON_STR_KEYS`, `OPT_SORT_KEYS`, `OPT_INDENT_2` semantics.
- Runtime env note: interpreter is **Python 3.14.4** (task brief states 3.13). Code uses `from __future__ import annotations`, `TimeoutError` builtin, `X | None` types — all compatible with 3.13 and 3.14. No version-specific breakage observed.
