### Note on memory_manager `_run_cleanups` iteration (lines 423-444)

This is the one prior MEDIUM that appears genuinely **STILL-PRESENT and unmitigated**: `_run_cleanups` iterates `self._cleanup_callbacks.items()` directly. If any registered cleanup callback calls `register_cleanup`/`unregister_cleanup` (directly or transitively) during its execution, CPython raises `RuntimeError: dictionary changed size during iteration`, which propagates out of the warning/critical cleanup path. The fix is a one-liner: iterate `list(self._cleanup_callbacks.items())`. Treat as LOW-MED (requires a self-modifying cleanup callback to trigger).

## Cross-cutting observations

1. **time.monotonic adoption is inconsistent.** `memory_manager.TTLCache` correctly switched to `time.monotonic()` (FIXED). `circuit_breaker`, `rate_limiter` bucket, and `shutdown_manager` budget still use `time.time()`. These are deadline/elapsed measurements and remain NTP/DST-sensitive. The prior audit's #4 recommendation is only partially adopted.

2. **The C1 gate has a hole on the hottest path.** Bulk-kill helpers are gated, but `ensure_single_instance` / `ensure_single_bot` (the function bot.py actually calls at startup) kills other bots and watchers directly without consulting `_kill_authorized`. Whether this is acceptable depends on intent (single-instance enforcement is the point), but it means "anyone who imports the module" can still trigger process kills via `ensure_single_bot(kill_existing=True)` — the prior C1 threat model is only partly closed. Recommend either gating it too or acquiring an OS lockfile so kills are coordinated, not racy.

3. **Sync-handler cancellation remains the biggest shutdown correctness gap.** Both `shutdown()` (executor futures) and `_atexit_handler` (daemon threads) cannot actually stop a stuck sync handler; they only stop *waiting* for it. This is now well-documented in code but unfixed.

4. **Verification confidence:** Findings are derived from reading every line and tracing logic; no source was modified. Library-behavior claims relied upon (executor futures non-cancellable, `gc.collect` holds the GIL, `asyncio.Lock` binds to the creating loop, `os._exit` vs `sys.exit` in signal handlers) are stable CPython semantics through 3.13 and are already correctly described in the code's own comments — no web verification altered any finding.
