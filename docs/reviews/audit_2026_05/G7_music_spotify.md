## Notes / Cross-cutting

### God-cog smell
`cogs/music/cog.py` is 2134 lines and embeds: queue management, voice lifecycle, persistence (DB + JSON), command handlers (~15), background tasks (cleanup, autosave), error handlers, presence updates, and owner permissions. Per the bot's own pattern (split out `spotify_handler.py`), a split would make sense:
- `voice.py` — connect/disconnect/move, on_voice_state_update, _auto_disconnect
- `playback.py` — play_next, fix, seek, after_playing callbacks
- `commands.py` — user-facing slash/text commands
- `persistence.py` — save_queue, load_queue, JSON fallback (delete this layer; consolidate with `queue.py:QueueManager`)
- `views.py` — already exists

### Duplication of QueueManager
`cogs/music/queue.py` defines a complete `QueueManager` class (313 lines) with proper locks and validation, but **the cog never uses it** — it manages queues directly via `self._guild_states[guild_id].queue`. The QueueManager exists but is unreachable code from the cog's path. Either delete it or rewrite cog.py to use it. Currently both implementations diverge subtly (lock semantics, JSON fallback paths, validation). The cog's path has the buggy load_queue (does not migrate to DB before deleting JSON, line 346); the QueueManager's path is correct.

### yt-dlp configuration is outdated for 2026
- `utils/media/ytdl_source.py` line 73: `"player_client": ["android", "web"]`. As of 2025-2026, Android and Web clients without PO Token frequently fail with HTTP 403. The current safer set is `["mweb", "tv"]` or `["ios"]`, plus PO Token integration via the `bgutil-ytdlp-pot-provider` plugin. Without PO Token, age-restricted and many regular videos fail. Recommend documenting cookies.txt as required, not optional.
- Line 50: `"source_address": "0.0.0.0"` — IPv4 binding workaround for IPv6 issues. With IPv6-only AWS / GCP VMs this *fails* to bind. Make conditional.
- Line 55: `max_filesize` exists, but yt-dlp's behavior on hitting this limit is to **delete the partial file silently**. If a track fails because it exceeds 300 MiB, user sees no clear "this track is too long" — it shows as a generic download error. Catch the specific case and message clearly.
- Line 65: `concurrent_fragment_downloads: 8` — aggressive; on YouTube's per-IP rate limits this triggers 429s faster than serial fragment downloads.
- No retry on `HTTP 429` specifically; yt-dlp's `retries: 5` covers 5xx but 429 is sticky.
- No `cookiesfrombrowser` option for Chrome/Firefox cookie auto-extraction (lower-friction than cookies.txt).

### Discord interaction 3-second rule
Several button callbacks call `interaction.response.send_message(...)` after sync work that may take milliseconds. None defer first. If discord.py adds latency on a slow server, the 3s window can elapse. Best practice: `await interaction.response.defer(ephemeral=True)` before any work, then `interaction.followup.send(...)` after.

### Voice client teardown
- `cog_unload` (line 197-208) calls `vc.disconnect(force=True)` which is correct for unload.
- `leave` (line 1495) and `fix` (line 1044) use `disconnect()` *without* `force=True` — these can hang up to 3s if Discord's voice WebSocket is laggy. Use `force=True`.
- No explicit `voice_client.stop()` before `disconnect()` in `leave` — discord.py handles this internally but explicit teardown prevents zombie audio threads.

### FFmpeg subprocess leaks
Five distinct paths construct `discord.FFmpegPCMAudio(...)`:
1. Line 653 (loop replay) — no `player.cleanup()` on construction-then-fail path
2. Line 730 (`from_url`) — `cleanup()` called at lines 786/799 on Discord/OS error
3. Line 1066 (fix) — no `player.cleanup()` if exception between construct and `play()`
4. Line 1776 (seek) — no `player.cleanup()` if `vc_seek.play(...)` raises (line 1813 except re-raises but doesn't cleanup the constructed player)
5. `YTDLSource.from_url` (utils/media/ytdl_source.py line 299) — wraps construction; if cls() raises after FFmpeg subprocess already started (e.g. PCMVolumeTransformer init), subprocess leaks

Each FFmpeg subprocess holds open file handles, network connections (for streaming), and ~10MB of memory per stream. Long-running bot will accumulate these.

### Presence change races multi-guild
Both the play_next path (line 854-860) and stop path (line 1451-1454) and auto_disconnect path (line 484-488) update `self.bot.change_presence(...)` based on `len(self.bot.voice_clients) <= 1`. With multiple guilds active, updating presence to "Listening to song A" in guild 1 while guild 2's track ends and resets presence to "คำสั่งเพลง" creates flickering. Use a global presence manager that sums across all music states.

### Per-guild lock granularity
`gs.play_lock` exists but is only acquired in `_play_next_once`. `stop`, `clear`, `shuffle`, `remove`, `seek`, etc. mutate `gs.queue` and `gs.current_track` without holding the lock. Either:
- Add a separate `state_lock` for non-playback state mutations
- Acquire `play_lock` before any state change (current play path may starve)

### yt-dlp blocking inside lock
The single biggest perf bug: `play_next` holds `play_lock` while doing `await YTDLSource.from_url(...)` (line 730) which can take 30+ seconds. During that time, every `!play` returns "another task is processing" (line 620 debug log). Re-architect: do the download outside the lock, only lock around the `voice_client.play(...)` call.

### Memory-leaking dicts / sets
- `queue.py:QueueManager._locks` (line 42): grows forever, never cleaned
- `cog.py:_queue_save_pending` (line 88): cleared on every periodic save (line 140) — fine
- `cog.py:_guild_states`: cleaned on guild_remove and cleanup_guild_data — fine

### Timeout / cancellation paths
`MusicControlView` timeout=180s. If a track is paused for >3 min, the buttons go disabled (on_timeout). Recommended: pass `timeout=None` for persistent views and register them via `bot.add_view(...)` in `cog_load`. Currently the views are not persistent so a bot restart loses all view state — buttons on old "Now Playing" embeds become non-functional after restart.

### Testing gaps
- No test verifies behavior when `cookies.txt` exists vs not
- No test for FFmpeg subprocess lifecycle on play() failure
- No test for the `play_next` lock-leak path (cancellation during shield)
- `test_music_cog_extended.py` exists per Grep but coverage of error paths unclear
