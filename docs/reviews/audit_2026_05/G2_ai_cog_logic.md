## Notes / Cross-cutting

### Architectural / design observations

1. **Dual ownership of state**: `ChatManager.pending_messages` and `cancel_flags` are aliased to `_message_queue.pending_messages` and `_message_queue.cancel_flags`. Many callers in `ai_cog.py` (lines 407-408, 439-440) reach into `chat_manager._message_queue.pending_messages` directly. The aliasing exists "for backward compatibility" per the comment, but the result is two access patterns (with the legacy alias and via the queue) that BOTH bypass the queue's `_queue_lock` (threading.Lock). External pop()s from these dicts are NOT thread-safe with the queue's internal mutations.

2. **RP-server hardcoded constants**: `GUILD_ID_RP`, `CHANNEL_ID_RP_COMMAND`, `CHANNEL_ID_RP_OUTPUT`, `GUILD_ID_MAIN`, `GUILD_ID_RESTRICTED`, `CHANNEL_ID_ALLOWED`, `GUILD_ID_COMMAND_ONLY` are all imported from `data.constants` and woven through both files as flow control. Adding a new server requires code changes. A per-guild config table would scale.

3. **Owner ID inconsistency**: Some commands use `@commands.is_owner()`, others use `if ctx.author.id != self.OWNER_ID` (where `OWNER_ID = CREATOR_ID` from env). discord.py's `is_owner()` walks `bot.owner_id`, `bot.owner_ids`, or `application_info().owner.id` — none of which need agree with the `CREATOR_ID` env variable. A single bot owner using the env var is fine; multi-owner setups will break.

4. **Single-character `!` prefix hardcoded** in the webhook proxy path (ai_cog.py line 575). The bot supports configurable prefixes (`_resolve_prefix_tuple`), but webhook-routed commands ignore that. Consequence: changing the prefix breaks Tupperbox/PluralKit users until code patch.

5. **Direct dict.del vs pop pattern**: The codebase has documented (and good) reasoning for using `pop(... , None)` instead of `del` to avoid races with the LRU evictor. But `link_memory` (line 988) and `move_memory` (line 1228, 1232) still use `del self.chat_manager.chats[target_id]`, which can KeyError under load.

6. **History format heterogeneity**: history items have `parts` that can be strings OR dicts with `"text"` or `"inline_data"`. Logic on lines 941-967 only handles string and `{"text": ...}` dicts, silently dropping other dict shapes. After process_attachments adds `{"inline_data": ...}` parts, those CAN be added to current_parts (line 884-893) but never replayed from history.

7. **Background task lifecycle**: `_background_tasks: set[asyncio.Task]` is a strong-ref pool. Tasks self-discard via `add_done_callback(self._background_tasks.discard)`. The set is NOT bounded; under burst load it can grow. Add a counter + warning log when len > 100.

8. **Mixin private-attribute coupling**: `SessionMixin`, `ResponseMixin` both reach into `ChatManager`-only attributes (`self.client`, `self.chats`, etc.). Stated contract is in the docstring, but the type annotations on lines 49-58 of session_mixin are class-level dummies (`bot: Bot`) that pyright will flag if mixin is used standalone. Acceptable in Python's duck-typing model, brittle in static analysis.

9. **Clear logging label**: "Gemini" appears in many error messages and metrics labels (`gemini_api`, `gemini_global`, `_call_gemini_api`, "Gemini Error") despite the bot being Claude-based. Confusing for ops.

10. **discord.py 2.7.1 API**: All Discord API uses are current for 2.7.x. No deprecated calls observed (no `discord.utils.snowflake_time` misuse, no `Client.run` with deprecated kwargs, no removed `Member.activities` patterns). `discord.utils.utcnow()` is used correctly.
