## Notes / Cross-cutting

### Circuit Breaker State Machine (circuit_breaker.py)

```
CLOSED ──(failure_count >= threshold)──> OPEN
OPEN ──(elapsed >= reset_timeout)──> HALF_OPEN  (auto, lazy in `state`/`can_execute`)
HALF_OPEN ──(any success)──> CLOSED  ⚠ first-success wins, see issue #156-158
HALF_OPEN ──(any failure)──> OPEN
```

Key issues with this state machine:
- **HALF_OPEN→CLOSED on first success is too eager.** With `half_open_max_calls=3`, three probe calls are admitted, but the first one to *complete successfully* closes the circuit. If probe #1 succeeds and probes #2, #3 fail, the circuit was already closed before #2/#3 reported — the failures then increment `_failure_count` against a fresh CLOSED state, requiring N more failures to re-open. This delays detection of a still-broken service.
- **Wall clock for deadlines.** All timestamps use `time.time()`. Recommend `time.monotonic()` throughout.
- **Per-instance, not per-key.** No per-user / per-endpoint circuit. Acceptable for a small service registry but doesn't scale to per-tenant isolation.

### Self-Healer Destructive Actions (self_healer.py)

**Every destructive action is auto-executed without explicit user confirmation when called from `auto_heal()` or the CLI.** The relevant calls:

4. `auto_heal(aggressive=True)` should be gated behind a separate API to prevent accidental triggering.
5. Add a `dry_run` mode that prints what *would* be killed.

### Async Concerns (cross-cutting)

- **`time.time()` vs `time.monotonic()`**: `circuit_breaker.py`, `rate_limiter.py`, `shutdown_manager.py` all use wall clock for deadlines. Per Python docs, monotonic is required for timeout/elapsed measurement to be NTP-safe.  See [PEP 418](https://peps.python.org/pep-0418/).
- **`async` methods that don't await**: `circuit_breaker.async_*`, `error_recovery._get_backoff_state_async`, `_reset_backoff_state_async`. These are misleading — they take a `threading.Lock` and return synchronously. Caller cannot rely on yielding.
- **`asyncio.Lock` per-key leaks**: rate_limiter.py acknowledges this, memory_manager.py doesn't have it but its `_lock`s are shared. No issue.
- **`asyncio.gather` not used for handler parallelism**: shutdown_manager runs handlers in series within a phase; for I/O-bound cleanup (DB close, HTTP client close) this is unnecessarily slow.

### Locks Held Across Awaits

Spot-checked all `with self._lock` (threading) and `async with lock` (asyncio) blocks for await statements inside:

- `circuit_breaker.py`: ✓ no awaits inside `_lock`.
- `error_recovery.py`: ✓ no awaits inside `_backoff_states_lock` or `ServiceHealthMonitor._lock`.
- `memory_manager.py`: ✓ no awaits inside TTLCache/`WeakRefCache` `_lock`.
- `rate_limiter.py`: `async with lock:` then `_get_or_create_bucket` (sync). No nested awaits while holding lock.
- `shutdown_manager.py`: `async with self._get_lock():` then sets phase. No nested awaits.

No deadlock-by-await-while-locked detected.

### Memory Manager Tuning (memory_manager.py)

Default `MemoryMonitor` thresholds (8 GB warning, 16 GB critical) are calibrated for "32GB DDR5 system" per the comment. **For a deployed Discord bot on a typical VM/container, these thresholds are 10-100× too high** — the OOM killer fires long before the monitor runs cleanup. Either:
1. Tune defaults to ~256MB warning / 512MB critical (typical bot footprint).
2. Read from env var `BOT_MEMORY_WARNING_MB` / `BOT_MEMORY_CRITICAL_MB`.
3. Auto-detect total system memory via `psutil.virtual_memory().total` and set thresholds as a percentage.

### Rate Limiter Concurrency (rate_limiter.py)

The most concerning section is `_get_or_create_bucket` lines 269-318:
- Eviction uses per-key locks (not global), so multiple coroutines can evict simultaneously.
- The "shared overflow bucket" branch downgrades per-user limits to a global limit silently — a DoS amplifier when an attacker holds many concurrent rate-limited operations open.

The acknowledged bugs in `cleanup_old_buckets` (lock leak, race conditions on lock recreation) suggest this module has been iterated on in response to bugs but hasn't been redesigned. A redesign with a single global `asyncio.Lock` for bucket management (with finer per-bucket logic underneath) would be simpler and correct.

### Shutdown Manager Signal Handling (shutdown_manager.py)

Per [Python signal docs](https://docs.python.org/3/library/signal.html) and [Super Fast Python on SIGINT](https://superfastpython.com/asyncio-control-c-sigint/):
- On Windows, SIGINT delivery to asyncio is unreliable; `loop.add_signal_handler` is **not supported**.
- Python 3.11+ added default cancellation of the main task on SIGINT, which obviates much of the manual handling.
- The current `_signal_handler → shutdown_sync → call_soon_threadsafe → ensure_future` chain is theoretically correct but the signal handler returns before shutdown completes — the OS may send SIGKILL if the signal is from a service manager (systemd, Windows Service Manager).

**Recommendation:** On Linux, use `loop.add_signal_handler` exclusively (the async path), and rely on Python 3.11+'s default SIGINT-cancels-main behavior on Windows. Document the Windows limitation.
