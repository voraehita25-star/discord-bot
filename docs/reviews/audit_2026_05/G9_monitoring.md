## Notes / Cross-cutting

### Two TokenTracker implementations (HIGH priority — duplication)

`utils/monitoring/token_tracker.py` (307 lines) and `cogs/ai_core/cache/token_tracker.py` (492 lines) are **two completely separate classes**, both named `TokenTracker`, with overlapping purpose and divergent feature sets:

| Concern | `utils/monitoring/token_tracker.py` | `cogs/ai_core/cache/token_tracker.py` |
|---|---|---|
| Sync/async | sync (`threading.Lock`) | async (`asyncio.Lock`) |
| Storage | in-memory dicts only | in-memory + SQLite `token_usage` table |
| Cost calc | none (just token counts) | per-model rates, USD output |
| Quota / limits | none (read-only stats) | hourly/daily user/channel/guild quotas with Thai error msgs |
| Eviction | LRU on count cap (`MAX_USERS=10000`) | trim-tail per key (`MAX_RECORDS_PER_KEY=5000`) |
| Time bucketing | local-time `%Y-%m-%d-%H` strings | tz-aware UTC datetimes via `_aware_now()` |
| Export | `export_stats()` JSON dict | none (relies on DB queries) |
| Persistence on restart | lost (in-memory only) | replayed via `init_from_db(hours=24)` |
| Imports | `from utils.monitoring.token_tracker import token_tracker` | `from cogs.ai_core.cache.token_tracker import token_tracker` |
| Test coverage | `tests/test_token_tracker.py` | `tests/test_cache_token_tracker.py` |
| Used by | `cogs/ai_core/imports.py:144` (re-exports), `tests/test_ai_integration.py` | `cogs/ai_core/ai_cog.py:182`, `cogs/ai_core/commands/debug_commands.py:359` |

**Both** have a module-level singleton named `token_tracker`. Importing both into the same scope shadows the previous one. `cogs/ai_core/imports.py` re-exports the **monitoring** one, while `ai_cog.py` and `debug_commands.py` import the **cache** one — meaning the codebase actually uses both simultaneously, recording the same usage data twice into different in-memory structures.

Also confusing: `utils/monitoring/token_tracker.py`'s `TokenUsage` dataclass has fields `input_tokens, output_tokens, request_count, timestamp` (a per-period bucket), while `cogs/ai_core/cache/token_tracker.py`'s `TokenUsage` has `input_tokens, output_tokens, timestamp, user_id, channel_id, guild_id, model, cached` (a single-event record). Same name, completely different meaning.

**Recommendation**: Promote `cogs/ai_core/cache/token_tracker.py` (the more featureful, async, DB-backed, quota-enforcing implementation) to the public monitoring API at `utils/monitoring/token_tracker.py`. Delete the simpler one. Migrate the two callers and the test file. Update the `imports.py` re-export.

### Three places define their own Sentry-style scrubber

1. `utils/monitoring/logger.py` `SensitiveDataFilter` (regex-based, applied to log records).
2. `utils/monitoring/sentry_integration.py` `_scrub_breadcrumb` (one-level dict scrub, calls `_redact_sensitive`).
3. `utils/monitoring/audit_log.py` `_scrub` (newline/CR strip, length cap).

The first two share the `_redact_sensitive` regex; the third is independent. Consider a single `utils.security.redact` module with `redact_secrets()`, `strip_log_injection()`, `truncate()` so all three call into one place.

### Health API surface is large and partially unauthenticated

The `/health/live`, `/livez`, `/health/ready`, `/readyz` endpoints are intentionally unauthenticated for k8s-style probes — but they leak `latency_ms`, `guilds`, server timestamp (with TZ offset). Trim the unauth payload to `{"status": "ok"}` only.

### Two HTTP servers, two ports

- `utils/monitoring/health_api.py` runs `ThreadingHTTPServer` on `HEALTH_API_PORT` (default 8080).
- `utils/monitoring/metrics.py` calls `prometheus_client.start_http_server` on its own port (default 8000).
- A third Go-side health API runs on `8082` reached via `health_client.py`.

Three monitoring HTTP servers in one process. Consider unifying behind one server (the `health_api.py` already exposes `/metrics`) and decommissioning the prometheus_client one — or document why both exist.

### Missing re-exports in `__init__.py`

`utils/monitoring/__init__.py` only re-exports `health_api` and `logger`/`structured_logger` symbols. Tests, the AI cog, and bot.py do deep imports for everything else. Either lock that in (and document it as the public API) or expand `__init__.py` to be the single import surface.

### Web / external research — current best practices

For comparison with current upstream:

- **Sentry SDK 2.x**: `before_breadcrumb` hook still works with the same signature; `push_scope` deprecated in favor of `new_scope`/`isolation_scope` (the code uses `new_scope` correctly). For `set_user_context`, **the v2 idiom is** `with sentry_sdk.isolation_scope() as scope: scope.set_user(...)` — fire-and-forget setting on the global scope is leaky (current behavior). `attach_stacktrace=True` with `with_locals=True` (default) leaks local variables; consider `with_locals=False` or write a `before_send` hook to sanitize stacktraces.
- **structlog**: more idiomatic for JSON structured logging in Python. The current `StructuredLogger` rolls its own; consider migrating to `structlog` for processor pipelines, formatter chaining, and built-in async support. Not blocking; current implementation is functional.
- **FastAPI/aiohttp health endpoints**: Current ThreadingHTTPServer is fine for low-volume. If health-check load grows, consider porting to `aiohttp.web` for the same event loop as the bot — eliminates thread/loop context switching. Best practice for k8s probes is to keep `/livez` and `/readyz` distinct (the current code does, with `/livez` checking process-alive and `/readyz` checking bot-ready) and to make them <100ms.
- **W3C Trace Context (OpenTelemetry)**: Use `traceparent` header (16 hex chars trace-id + 16 hex chars span-id + 2 hex chars flags) for cross-service tracing. Current `X-Trace-ID` is non-standard.
