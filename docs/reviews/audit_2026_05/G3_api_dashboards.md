## Notes / Cross-cutting

### Severity inflation
Several issues marked HIGH require chained conditions to exploit (e.g. malicious frontend + prompt injection in attached document). Treated as HIGH because:
- The CLI backend's prompt-injection vector via Read tool combined with broad env-var inheritance is a credential-disclosure risk on a single-user dashboard that nonetheless runs with the full bot permissions.
- The `pypdf >=5.0.0` pin is a definite outdated dependency with multiple confirmed DoS CVEs in the 5.x line.

### Architectural observations
1. **Three near-identical chat handlers** (`dashboard_chat.py`, `dashboard_chat_claude.py`, `dashboard_chat_claude_cli.py`) — total 3,689 LOC — share extensive logic that is not extracted: prompt building, history loading, image preprocessing, retry loops, token estimation, search/replace patching. Refactor into a `BaseChatHandler` base class + per-provider methods.
2. **`conversation_id` validation** is duplicated in 9+ places, none with a length cap. Centralize as `validate_conversation_id(s) -> bool` in `dashboard_common.py`.
3. **Cache invalidation** for `_USER_CONTEXT_CACHE` is scattered across `handle_save_memory`, `handle_delete_memory`, `handle_save_profile`, `handle_delete_document_memory`, `handle_update_document_memory`. Easy to miss when adding new mutation paths.
4. **API failover** is integrated into the SDK Claude path but not the CLI path (`_FAILOVER_AVAILABLE` is hard-skipped). When CLI is enabled but the user wants both subscription AND API as fallback, no path exists.
5. **No per-WS connection limit by IP/origin** — only global `MAX_CLIENTS=20`. A single IP can claim all 20 slots.
6. **Streaming backpressure**: `ws.send_json` is awaited but if the client's TCP receive buffer fills, the await blocks. Concurrent sends to many clients can stall the entire event loop — confirmed in `_on_endpoint_changed`'s serial loop.

### Known-good practices
- `expected_conversation_id` parameter on DB updates (defense in depth against cross-conv tampering) — well-applied.
- `hmac.compare_digest` for token comparison — correct.
- Strict TLS gate on non-localhost bind (`dashboard_config.py:30–42`) — well-enforced at config load.
- `LeadingTimestampStripper` for streaming output — clever, well-tested in `dashboard_common.py` tests.
- Per-conversation lock for CLI subprocess serialization (line 232–255).
- `_failover_retry` documented as server-internal-only (`dashboard_chat_claude.py:185–194`).

### Suggested follow-up tickets
1. Pin pypdf>=6.8.0 in requirements.txt.
2. Add zip-bomb protection around `docx.Document(...)` parse.
3. Gate CLI backend on `DASHBOARD_ALLOW_UNRESTRICTED` env var.
4. Allowlist subprocess env for CLI backend; strip Discord/DB/etc. secrets.
5. Add `--add-dir` scoping to Claude CLI argv to limit Read-tool reach.
6. Length-cap `conversation_id` to 128 chars at every entry point.
7. Length-cap pre-auth WS frames to 4 KiB.
8. Replace `dict` LRU cache with `OrderedDict.move_to_end` for true LRU semantics.
9. Drain stderr concurrently with stdout in `_run_claude_subprocess`.
10. Move duplicated `_apply_search_replace` and `_retry_delay_seconds` into a shared module.
11. Per-IP failed-auth backoff before WS upgrade.
12. Refactor 3 chat handlers into a base class.
