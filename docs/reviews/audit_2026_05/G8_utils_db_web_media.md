## Notes / Cross-cutting

### SSRF posture summary

**Two parallel implementations exist:**
1. `utils/web/url_fetcher.py` — primary, has `_SSRFSafeResolver` wrapper (good defense-in-depth against DNS rebinding) and explicit `_BLOCKED_NETWORKS`.
2. `utils/web/url_fetcher_client.py` — fallback path duplicates the SSRF check using `addr.is_private/is_loopback/...`. Lists drift.

**Confirmed gaps** (in order of risk):
- **IPv4-mapped IPv6** (`::ffff:127.0.0.1`) bypasses both checks because the `127.0.0.0/8` entry is an IPv4 network and the v6 entries don't match the mapped form. **Exploitable**.
- **Scheme allowlist** missing in `url_fetcher.py`'s `_is_private_url`; redirect targets only blocked by IP. `gopher://`, `dict://`, `ldap://` may bypass. `url_fetcher_client.py` enforces this in the fallback path; `url_fetcher.py` does not.

- **Two SSRF check copies** will diverge over time — single source of truth needed.

### SQL/Database posture summary

- All user-input parameters are bound (`?`) — no string concatenation found that includes user data.
- Two f-string SQL paths (`f"UPDATE ... SET {set_clause} ..."` and `f"SELECT * FROM [{table}]"`) both use whitelist + identifier regex + bracket quoting. **Acceptable**, but defense-in-depth.
- WAL mode set in `init_schema` (line 290), `wal_autocheckpoint=2000` (line 292), per-connection `synchronous=NORMAL` (line 979/1220), `cache_size` 250000/100000 pages, `temp_store=MEMORY`. Periodic `wal_checkpoint(TRUNCATE)` every 10 min.
- `foreign_keys=ON` re-asserted on every connection acquisition (good — comment at line 983-988 explains why migration 010 toggled it OFF).
- Connection pool: BoundedSemaphore(32) + Queue(maxsize=16). Pool replacement during reinit is partially racy (queue replaced; semaphore mutated in place — inconsistent).
- Write lock prevents SQLITE_BUSY within the process; multi-process deployments need `PRAGMA busy_timeout=N` (currently relies on `aiosqlite.connect(timeout=30)`, which sets `sqlite3` busy timeout — OK).
- **No `PRAGMA busy_timeout` set explicitly** — relies on the connect-time default from aiosqlite. Should be made explicit (e.g. `PRAGMA busy_timeout=30000`) for clarity.

### Migration system posture summary

- Versioned, sequential, tracked in `schema_version`. Backups before applying. Last-5 retention.
- **No checksum verification on previously-applied migrations** — drift goes undetected.
- Statement splitter is naive (line-by-line + `complete_statement`) — would fail on triggers or compound BEGIN…END blocks.
- `init_schema` ad-hoc ALTERs duplicate migration 014's documented ALTERs — confusing.
- **`KNOWN_TABLES` (`utils/database/__init__.py`)** out of sync with current schema — `dashboard_document_memories` and `dashboard_conversation_tags` are silently dropped from JSON exports.

### orjson divergence

`utils/fast_json.json_dumps` accepts `**kwargs` for compat with stdlib `json.dumps`, but orjson branch silently drops `default=`, `sort_keys=`, `cls=`, etc. Bot's code mostly uses stdlib `json` directly (database.py line 240, 369, 711, etc.) so the divergence isn't biting today — but anyone migrating to `fast_json.json_dumps(obj, default=str)` will hit `TypeError` from orjson on `datetime`/`UUID`. Either translate kwargs or drop them from the signature.

### Localization safety

User input flows through `.format(**kwargs)` — no eval/exec/template-injection vector here. Localized strings are sent to Discord which handles markdown safely. No sanitization needed for current usage; document the constraint if any localized string is ever piped to HTML/shell.

### General code-quality observations

- Deeply nested try/except blocks in database.py (especially `_reinitialize_pool` and `get_connection_with_retry`) make control flow hard to verify by hand. Refactor to smaller functions with explicit cleanup.
- `import json` and `import time as _time` repeated inside many methods — should be module-level.
- Several `from urllib.parse import urlparse` imports inside functions (url_fetcher.py:154, 271, ytdl_source.py:218, 315, url_fetcher_client.py:152) — module-level once.
- `from utils.monitoring.metrics import metrics` repeated inside hot paths (database.py:912, 943, 1024) — guard with `try/except ImportError` once at module level.
