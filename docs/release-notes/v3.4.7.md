# v3.4.7 — Max-capability tuning: deeper recall + lifted limits

**Date:** 2026-06-13

Capability release. The generation engine was already at full power (Opus 4.8,
`max_tokens` 128000, effort `xhigh`); this release lifts the **memory / context**
throttles that were the real quality bottleneck, then raises the rate limits and
removes the token-quota system for a small self-hosted deployment. No platform or
security constraints were weakened (Discord's 2000-char cap and the
injection/mention sanitization stay). Every suite stays green.

## Deeper recall & context (env-tunable)

- **RAG retrieval 3 → 15 per turn** (`RAG_TOP_K`) and **entity retrieval 3 → 8**
  (`ENTITY_TOP_K`). The production path was hard-capped at 3 long-term memories
  despite the 1M-token window — recall was the dominant bottleneck.
- **In-context history 2000 → 8000 messages** (`MAX_HISTORY_ITEMS`); the CLI path's
  token-trim + 1.2M-char ceiling still keep it within the window.
- **Dashboard history window 100 → 500** (`DASHBOARD_HISTORY_MESSAGES`).
- **Semantic embeddings decoupled from `CLAUDE_BACKEND`** — Gemini embeddings now
  turn on whenever `GEMINI_API_KEY` is set (works on the default `cli` backend too);
  `RAG_EMBEDDINGS=off` to force keyword-only. Keyless setups behave exactly as before.
- **Memories persist even without an embedding** — `add_memory` now saves a
  vectorless (keyword-searchable) row instead of silently dropping it; the FAISS
  index skip-guards keep the semantic index clean.
- **Summarizer keeps 50 → 200 recent messages** and a richer 300 → 1000-token
  summary (api backend); **streaming forwards `effort`** and widens the stall
  timeouts (30→120 / 10→45 s) so slow, thoughtful replies aren't truncated.

## Rate limits raised (small trusted group)

Per-user/guild AI limits raised for a ~5-person deployment while keeping the
token-bucket + adaptive (circuit-breaker-aware) throttle that guards against
Anthropic 429s and runaway loops:

| Limit | Before → After |
|---|---|
| `ai_user` (per user) | 10 → 60 / min |
| `ai_guild` (per guild) | 30 → 240 / min |
| `gemini_api` (per user) | 15 → 60 / min |
| `gemini_global` | 60 → 300 / min |
| `ai_tool_call` (per user) | 5 → 30 / min |
| `command` / `spam` / `server_management` | 5→20 / 3→10 / 10→30 |
| Dashboard WS messages/client | 30 → 120 / min |

## Token quota removed

The per-user/channel/guild **token-quota enforcement** is gone entirely: the
`UsageLimits` caps, the `check_limits()` gate, and the `AI_TOKEN_QUOTA_ENFORCE`
toggle are removed. The bot no longer blocks any request on a token budget (it
already ran on a Max subscription with no per-token billing). Token **usage
tracking/analytics** is retained — `record_usage`, stats, and cost estimation
still feed the dashboard's cost display; they just never block a request.
