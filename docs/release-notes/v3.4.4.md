# v3.4.4 — Dashboard start-progress: no more false "start timed out"

**Date:** 2026-06-04

Patch release fixing a long-standing false alarm in the native dashboard: clicking
**Start** on a slow cold start surfaced a spurious *"Bot start timed out — check
logs"* warning even though the bot was booting normally and came up fine moments
later. The fix replaces the frontend's fixed 15-second deadline with a real,
backend-reported start-progress signal — so a slow-but-healthy start never warns,
and a genuine startup failure is now reported *faster and more accurately*, with the
process exit code. No breaking changes; no Python or Go changes.

## The bug

`bot.py` only writes its `bot.pid` file (the dashboard's "running" signal) *after* a
heavy import + startup-check phase. On a cold start (post-reboot, antivirus scanning
the `.pyd`/`.dll` extensions, busy disk) that phase can take well over the
frontend's hard-coded 15s poll deadline, so `waitForRunning` gave up and fired a
`warning` toast — a false alarm on every slow start. The flat poll had only two
outcomes ("running" or "timed out") and could not tell a slow boot apart from a real
crash; both collapsed into the same misleading warning.

## The fix — state-aware start detection

- **Backend (`native_dashboard/src/bot_manager.rs`)** — new `StartProgress` enum and
  `start_progress()` method that pair the PID-file signal with the liveness of the
  `Child` handle the manager already holds from `start()`. It resolves to one of four
  states: `running` (pid written + cmdline verified), `starting` (spawned process
  still alive, mid-boot), `exited{code}` (process died before becoming ready — a real
  failure), or `unknown` (no tracked startup child).
- **New Tauri command `get_start_progress`** (`src/main.rs`) exposes it to the UI and
  is registered in the invoke handler.
- **Frontend (`src-ts/app.ts`)** — `waitForRunning` is replaced by `waitForStart`,
  which polls `get_start_progress`: it resolves "Bot started" the instant the PID
  lands, surfaces "Bot failed to start (exit code N) — check logs" *immediately* on a
  real crash, keeps waiting (with a soft "cold start" notice at 12s) while the process
  is alive, and — only if a boot hangs past 60s — hands back to the periodic status
  refresh with an informational message instead of a false warning. The dead
  `'stopped'` branch of the old helper was removed.
- `src-ts/types.ts` gains the `StartProgress` discriminated-union type (mirrors the
  Rust enum's internally-tagged serde shape); the e2e Tauri mock
  (`tests-e2e/_fixtures/mock-tauri.ts`) gains a `get_start_progress` default so a
  Start click can't hang a headless test.
